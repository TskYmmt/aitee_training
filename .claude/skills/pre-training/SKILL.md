# pre-training

応用情報技術者試験のテキストを 1 leaf 単位で能動学習させるスキル。Layer 1（Pre-training）の実装。learner の脳内に retrievable な表現を作ることが目的で、過去問を解かせる Layer 2（SFT）とは別物。

## このスキルの役割

learner が「Xを勉強したい」と言ったときに、X の概念表現を強制的に作る。教科書を**読ませる**のではなく、**説明→問題→フィードバック**のループを回して engagement を強制する。受動的な読書では「目が滑る」現象で実効学習率がゼロに近づくため、必ず能動的な処理を要求する構造にする。

## 入力

learner の曖昧な学習対象指定をそのまま受け取る:

- leaf ID（例: `ch04_03_01`）
- 節番号（例: `4-3-1`、`4-3`）
- タイトルの一部（例: 「デュアルシステム」「信頼性」）
- 章名（例: 「4章」「ネットワーク章」）

learner が形式を意識しなくていいように、何が来てもこのあと textbook-dispenser に丸投げする。

## 必須の前処理: textbook-dispenser を呼ぶ

learner の入力を必ず textbook-dispenser スキルに通して、厳密な leaf を解決する。直接 textbook_structure.json を読まないこと。

```bash
python3 /Users/tskymmt/git/aitee_training/.claude/skills/textbook-dispenser/scripts/dispense.py resolve "<learner input>"
```

返ってきた JSON の `status` で分岐:

- `ok`: そのまま leaf 学習に進む
- `ambiguous`: `candidates` リストを learner に番号付きで提示し、選んでもらう
- `not_found`: `hint` を learner に伝えて再入力を促す
- `too_many`: 絞り込みを促す

learner が候補から選んだら、その leaf ID で `lookup` を呼んで詳細情報（隣接 leaf 込み）を取得してから Phase 1 に進む。

## 1 leaf エピソードの構造（必ずこの順序）

learner と learning 対象 leaf が確定したら、以下のフローを 1 セットとして回す。

### Phase 1: Concept exposition（説明）

leaf のタイトルと位置を 1 行で示してから、**密度の高い箇条書き**で内容を説明する。

含めるべき要素:

- **定義**: その概念が何か（1〜2行）
- **特徴・性質**: 重要なポイントを 5〜10 個、短い箇条書きで
- **用途・適用場面**: どこで使うか
- **隣接概念との違い**: 似た用語との対比（試験で問われやすい）
- **具体例**: あれば 1〜2 個

冗長な前置き（「以下に説明します」「重要な概念を学びましょう」等）は不要。試験で問われるレベルの粒度で、簡潔に。

### Phase 2: Bridge（隣接概念への予告）

次の leaf や対比される類似概念があれば、軽く前振りしておく:

> 「次に学ぶデュプレックスシステムとの違いを意識しておくと理解が深まります」

これで learner の脳内で次概念へのリンクを早めに張る。1〜2 行で十分。

### Phase 3: 10〜20 問の engagement loop

問題を **1 問ずつ** 提示する。一度に複数問を出さない。learner の答えを受け取ってからフィードバックし、次の問題に進む。

#### 問題数

- 用語が少なめ・概念がシンプル → **10 問**
- 用語が多い・対比が複雑・コラム的に深い → **20 問**
- 通常 → **15 問前後**

leaf の情報密度に応じて柔軟に決める。

#### 問題のジャンル（必ず混ぜる）

同じ概念を異なる角度から retrieve させることが目的。1 leaf エピソードの中で以下のジャンルを混在させる:

1. **定義想起**（Q1〜Q3 程度）— 「Xとは何か」「Xを表す用語はどれか」
2. **特徴の識別** — 「Xの特徴として正しいものはどれか」
3. **対比・識別境界** — 「XとYの違いは何か」「次のうちXに該当するのはどれか」
4. **シナリオ適用** — 「〜という要件のときに最適なのはどれか」
5. **誤り発見**（Q9〜Q10 あたり、最終問題に近い位置）— 「次の記述のうち誤っているものはどれか」（複数記述から1つの間違いを発見させる）

このジャンルミックスで、leaf の内容が「定義／機能／使い分け／典型的な誤解」の全角度からカバーされる。

#### 形式

基本は **4 択（A, B, C, D）**。learner が答えやすいリズムを優先する。記述問題は使わない（フィードバックループが遅くなり engagement が落ちる）。

問題の提示形式の例:

```
問題 3／15
デュアルシステムにおいて、2つのシステムの処理結果を比較する装置を何と呼ぶか？

A. ロードバランサ
B. コンパレータ（照合機）
C. アキュムレータ
D. マルチプレクサ
```

「問題 N／合計」を必ず先頭に置く（learner が進捗を体感できる）。

#### フィードバック（毎問ごと、これが本体）

learner の回答を受け取ったら、必ず以下を含むフィードバックを返す:

1. **正誤**: 「正解！」または「正解はBです」
2. **なぜ**: 正解の理由を 1〜2 行
3. **間違い選択肢の正体**: 不正解の選択肢が「実は何の概念か」を簡潔に示す。これが重要 — 1 問で複数概念に触れるレバレッジになる
4. **覚え方やニーモニック**: あれば追加（例: 「ホットは熱い＝動いている」）
5. **過去の関連問題への言及**: 同じ leaf 内の前問と関連していれば一言

フィードバックは「答え合わせ」ではなく「learner の脳内 concept graph にエッジを増やす機会」として書く。短く、ただし密度を持って。

フィードバック例:

```
正解！

コンパレータ（照合機）は、2つのシステムの出力を常に比較して、一致／不一致を判定する装置です。
不一致が検出されたら障害発生と判断します。

ちなみに他の選択肢の正体:
- ロードバランサ → 負荷分散装置
- アキュムレータ → CPU内の演算結果を一時保持するレジスタ
- マルチプレクサ → 複数の入力から1つを選択して出力する回路
```

### Phase 4: Episode summary

15 問前後が終わったら、leaf の要点を 3〜7 個の箇条書きで再掲する。learner の脳内で leaf 全体を一度圧縮させる。

例:
```
ここまでのまとめ:
- デュアルシステム → 2台で同じ処理を並行実行＋照合、信頼性最高、コスト2倍
- 結果照合は「コンパレータ」が担当
- 障害時は故障側を切り離して縮退運転（フェールソフト）
- 用途は銀行勘定系・航空管制などミッションクリティカル
```

### Phase 5: 次の leaf への遷移（learner consent gated）

明示的に learner に確認する:

> 「次は ch04_03_02『デュプレックスシステム』に進みますか？」

learner の同意を得てから次の leaf に進む。**自動で次に進まない**。learner には常に「ここで止まる」「次に進む」「他の leaf に飛ぶ」「休憩する」の選択権がある。

learner が了承したら、textbook-dispenser の `neighbors` を呼んで次の leaf を取得し、Phase 1 から繰り返す:

```bash
python3 /Users/tskymmt/git/aitee_training/.claude/skills/textbook-dispenser/scripts/dispense.py neighbors <current_leaf_id>
```

learner が「ちょっと休憩」「今日はここまで」と言ったら、ここまで完了した leaf を一覧で報告して終了する。

## 守るべきこと

- **問題は 1 問ずつ提示**: 複数同時提示しない。1 問に集中させるリズムを守る
- **必ずフィードバックを返す**: 正誤だけでなく「なぜ・他選択肢の正体・関連」を含める
- **leaf 単位で完結**: 1 leaf を始めたら途中で別 leaf に飛ばない（learner が明示的に望まない限り）
- **遷移は learner consent**: 自動で次に進まない。「進みますか？」を必ず聞く
- **column も leaf として扱う**: dispenser が `is_column: true` を返しても通常の learning フローに乗せる
- **textbook_structure.json は直接読まない**: 必ず textbook-dispenser を経由する

## してはいけないこと

- **一度に全問を出す**: NG。learner の集中が分散し engagement が落ちる
- **説明を省いて問題に行く**: NG。先に exposition がないと engagement の足場がない
- **フィードバックを 1 行で済ます**: NG。フィードバックの密度がこのスキルの本体
- **過去問を Layer 1 として使う**: NG。過去問は Layer 2 (SFT) のデータ。pre-training では「概念を多角的に retrieve させる engagement 問題」を新規生成する
- **学習進捗を勝手に保存する**: NG。このスキルは状態を持たない。記録は learner または上位レイヤの責務

## なぜこの構造か（補正2 の実装）

人間脳では受動的な露出（教科書を読むだけ）では実効学習率がゼロに近い（「目が滑る」現象）。pre-training フェーズでも engagement を強制する仕掛けが必要で、説明→問題→フィードバックのループはその実装。問題は「テスト」ではなく「engagement enforcer」として機能する。learner が答えを書く瞬間に必ず forward pass が走り、フィードバックが正解を上書きしつつ間違い選択肢の正体を通して横方向のエッジを張る。これが 1 leaf あたり 15 回繰り返されることで、単なる読書では作れない密度の representation ができあがる。

このスキルは Layer 1（Pre-training）専用。Layer 2（SFT、試験フォーマット適応）と Layer 3（Maintenance、間隔反復）は別スキルが担う。
