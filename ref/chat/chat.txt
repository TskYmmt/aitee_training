4-3-1 デュアルシステム構成
デュアルシステムとは、同じ処理を2つのシステム（CPU）で同時に並行して実行し、その結果を**照合（クロスチェック）**する構成です。

ポイントをまとめると：

2つのシステムが常に同じ処理を行う
処理結果を**照合機（コンパレータ）**で比較する
結果が一致すれば正常、不一致なら障害と判断
障害が検出されたら、故障側を切り離して片方だけで処理を継続できる
信頼性は非常に高いが、コストが約2倍かかる
主な用途：銀行の勘定系、航空管制など絶対に止められないミッションクリティカルなシステム
デュプレックスシステムとの違い（次で詳しくやりますが先に軽く）： デュアルシステムは「2台とも常に同じ仕事」、デュプレックスシステムは「1台が本番、もう1台は待機」という違いがあります。

4-3-2 デュプレックスシステム
デュプレックスシステムとは、本番系（主系・現用系）と待機系（従系）の2台で構成し、普段は本番系だけが処理を行い、障害時に待機系に切り替える構成です。

待機系の種類（ここが超重要！）
ホットスタンバイ

待機系は電源ON・OSやアプリも起動済みの状態で待機
障害時に即座に切り替え可能（切替時間が短い）
待機中は基本的に何もしていない（または同期をとっている）
コストは高めだが復旧が速い
ウォームスタンバイ

待機系は電源ON・OSは起動済みだが、アプリは未起動
切替にはアプリの起動時間が必要
ホットとコールドの中間
コールドスタンバイ

待機系は電源OFFの状態
障害時にOS起動から始めるので切替に時間がかかる
コストは最も安い
待機系を普段はバッチ処理など別の業務に使えることもある
デュアルシステムとの比較
デュアル → 2台が同じ処理、常に照合、コスト高・信頼性高
デュプレックス → 1台が本番、1台が待機、コスト抑えめ・切替時間が発生
では問題いきます！

4-3-3 災害を想定したシステム構成
地震・火災・洪水などの大規模災害が起きた場合、同じ建物にあるシステムは本番系も待機系もまとめてダウンする可能性があります。そこで地理的に離れた場所にバックアップサイトを用意する考え方が重要です。

バックアップサイトの種類
ホットサイト

本番と同じ機器・データが常に同期されている遠隔拠点
災害時に即座に切替可能
コスト最高だが復旧時間（RTO）が最短
ウォームサイト

機器は設置済みだが、データの同期は定期的（最新ではない）
切替にはデータの復旧作業が必要
コスト・復旧時間ともに中間
コールドサイト

建物・電源・回線など最低限のインフラだけ用意
機器の搬入・設置・データ復旧すべて必要
復旧に数日〜数週間かかる
コスト最安
関連用語
BCP（事業継続計画）

災害時でも重要な業務を継続・早期復旧するための計画
RTO（目標復旧時間）

システムを復旧させるまでの目標時間。短いほど高コスト
RPO（目標復旧時点）

どの時点のデータまで復旧できるか。RPOが0に近い＝データロスが少ない
スタンバイ方式と似た構造（ホット/ウォーム/コールド）なので、対比で覚えると楽です。では問題いきます！

RPOはRecovery Point Objectiveで、「どの時点のデータまで復旧できるか」を示します。

RPOが0に近い → 災害直前のデータまで復旧できる → データロスがほぼない
RPOが24時間 → 最大24時間分のデータが失われる可能性がある
Aの「復旧にかかる時間が短い」はRTOの話ですね。

RTO → Time（時間）→ どれだけ早く復旧するか
RPO → Point（時点）→ どの時点のデータまで戻せるか

4-3-4 高信頼化システムの考え方
システムの信頼性を高めるための設計思想をまとめたセクションです。用語が似ていて紛らわしいので、しっかり区別しましょう。

フォールトトレランス（fault tolerance）
障害が発生しても正常に動作し続けることを目指す考え方全体
デュアルシステムやデュプレックスシステムもこの一種
フェールソフト（fail soft）
障害時に一部の機能を縮小してでも全体の稼働を継続する
別名：縮退運転（デグレード）
例：デュアルシステムで片方を切り離して1台で継続
フェールセーフ（fail safe）
障害時に安全な状態に移行する
例：信号機が故障したら**赤（停止）**にする
例：ストーブが転倒したら自動消火する
フェールオーバー（failover）
障害時に待機系に自動で切り替えること
デュプレックスシステムで使われる
フールプルーフ（fool proof）
人間が誤操作しても問題が起きないようにする設計
例：電子レンジはドアを閉めないと動かない
例：USBは向きが合わないと挿せない（※Type-C以前）
フェールバック（failback）
フェールオーバー後、元の本番系に戻すこと
似た名前が多いですが、それぞれ「何を守るか」が違います。では問題いきます！

4-4 高信頼性・高性能システム
4-4-1 クラスタリングとクラスタシステム
クラスタリングとは、複数のコンピュータを連結して1つのシステムのように動作させる技術です。

クラスタシステムには大きく2種類あります：

HAクラスタ（高可用性クラスタ）

可用性（止まらないこと）を重視した構成
1台が故障しても他のサーバが処理を引き継ぐ
フェールオーバーの仕組みを使う
例：Webサーバの冗長化
HPCクラスタ（高性能計算クラスタ）

処理性能の向上を重視した構成
複数のサーバで処理を分散・並列実行して高速化
例：科学技術計算、気象シミュレーション
4-4-2 グリッドコンピューティング
ネットワーク上の多数のコンピュータの遊休資源（空いているCPUなど）を集めて、巨大な計算能力として活用する技術
クラスタリングとの違いは、地理的に分散した異種のコンピュータも参加できる点
例：SETI@home（宇宙人探しプロジェクト）、タンパク質解析
コラム：ロードバランサ（負荷分散装置）
複数のサーバに処理を均等に振り分ける装置
特定のサーバに負荷が集中するのを防ぐ
HAクラスタと組み合わせて使われることが多い
では問題いきます！

4-5 ストレージ関連技術
4-5-1 RAID
RAIDとは複数のディスクを組み合わせて、信頼性や性能を向上させる技術です。RAID（Redundant Arrays of Inexpensive Disks）の略で、「安価なディスクの冗長配列」という意味です。

主要なRAIDレベルを押さえましょう：

RAID 0（ストライピング）

データを複数のディスクに分散して書き込む
読み書きが高速になる
ただし冗長性なし。1台壊れたら全データ消失
信頼性は上がらない、速度だけ上がる
RAID 1（ミラーリング）

同じデータを2台のディスクに同時に書き込む
1台壊れてももう1台にデータが残る
信頼性は高いが、容量効率は50%（2台で1台分の容量）
RAID 5（パリティ分散）

データと**パリティ（誤り訂正用の情報）**を複数ディスクに分散
最低3台必要
1台故障してもパリティからデータ復元可能
容量効率が良く、実用で最もよく使われる
RAID 0+1 / RAID 1+0（RAID 10）

RAID 0とRAID 1を組み合わせた構成
高速性と信頼性の両立
最低4台必要、コスト高
4-5-2 ストレージの接続形態
DAS（Direct Attached Storage）

サーバに直接接続するストレージ
シンプルだが共有が難しい
NAS（Network Attached Storage）

LANに接続してファイル単位で共有するストレージ
ファイルサーバのイメージ
導入が容易で中小規模向き
SAN（Storage Area Network）

ストレージ専用の高速ネットワークを構築して接続
ブロック単位でアクセス（高速）
大規模システム向き、コスト高
コラム：Hadoop
大量のデータを分散処理するためのフレームワーク
ビッグデータの処理で活用される
では問題いきます！

4-6 仮想化技術
4-6-1 サーバ仮想化
サーバ仮想化とは、1台の物理サーバ上に複数の仮想サーバ（仮想マシン）を動作させる技術です。

メリット

サーバの統合・集約によりハードウェアコスト削減
リソース（CPU・メモリ）の有効活用（1台のサーバの余剰資源を無駄にしない）
仮想マシンの追加・削除が柔軟にできる
異なるOSを1台の物理サーバ上で同時に動かせる
主な実現方式

ホスト型

通常のOS（ホストOS）の上に仮想化ソフトを載せ、その上で仮想マシンを動かす
導入が簡単だがオーバーヘッド（余計な処理負荷）が大きい
例：VirtualBox、VMware Workstation
ハイパーバイザ型

ホストOSなしで、ハードウェア上に直接ハイパーバイザを載せる
その上で仮想マシンを動かす
オーバーヘッドが小さく高性能
企業の本番環境で主流
例：VMware ESXi、Hyper-V、KVM
コンテナ型

仮想マシンではなく、OS上でアプリの実行環境（コンテナ）を隔離する
カーネルをホストOSと共有するので軽量・高速起動
例：Docker、Kubernetes
4-6-2 ストレージ仮想化
複数の物理ストレージを1つの大きな仮想ストレージとして見せる技術
シンプロビジョニング：実際に使う分だけ物理容量を割り当て、見かけ上は大きな容量を割り当てる。容量の無駄を削減
ライブマイグレーション：仮想マシンを稼働したまま別の物理サーバに移動できる技術
では問題いきます！

ここで「あれ、それってホスト型と似てない？」と思うかもしれませんが：

ホスト型 → ホストOS上で**仮想マシン（個別OSあり）**を動かす → 重い
コンテナ型 → ホストOS上でアプリの実行環境だけ隔離 → 軽い
コンテナは「OS丸ごと」ではなく「アプリだけ箱に入れる」イメージです。だからめちゃくちゃ速く起動できるんですね。

ハイパーバイザってことはOS上ではない？

3月27日
そのとおりです！

EC2はホストOSなしで、ハードウェアの上に直接ハイパーバイザ（Nitro）が載っています。

ホスト型との違いをもう一度整理すると：

ホスト型（VirtualBox等） → 自分のPC上に「Windows → VirtualBox → 仮想マシン」と載せる。ホストOSがある
ハイパーバイザ型（EC2等） → 物理サーバに「ハイパーバイザ → 仮想マシン」と載せる。ホストOSがない
AWSのデータセンターにある物理サーバは、普通のWindowsやLinuxが入っているわけではなく、Nitroハイパーバイザだけが入っていて、その上にユーザーのEC2が大量に動いている、というイメージです。

デュアルブート

トランザクションてなに

3月27日
トランザクションとは、分けることのできない一連の処理のまとまりのことです。

一番わかりやすい例が銀行の振込みです：

AさんからBさんに1万円を振り込む

この処理は内部的に2つのステップがあります：

Aさんの口座から1万円を引く
Bさんの口座に1万円を足す
この2つは必ずセットで成功しないといけません。もし1だけ成功して2が失敗したら、1万円が消えてしまいますよね。

だからこの2つをまとめて1つのトランザクションとして扱い：

両方成功 → コミット（確定）
どちらか失敗 → ロールバック（全部取消して元に戻す）
という仕組みにしています。

トランザクションの4つの性質（ACID特性）
A（原子性） → 全部成功か全部取消、中途半端なし
C（一貫性） → 処理前後でデータの整合性が保たれる
I（独立性） → 他のトランザクションの影響を受けない
D（永続性） → 確定した結果は障害が起きても消えない
これはデータベース（第6章）で詳しくやりますが、今の段階ではTPCの「トランザクション処理」＝「このような一連の処理を1秒に何件さばけるか」を測るもの、と理解しておけばOKです。

4-8 待ち行列理論の適用
計算問題が出るセクションです。考え方をしっかり押さえればパターンで解けます。

4-8-1 待ち行列理論の基本事項
待ち行列とは、サービスを受けるために並んで待つ現象をモデル化したものです。コンビニのレジやコールセンターと同じイメージです。

基本用語

到着率（λ：ラムダ） → 単位時間あたりに到着する要求の数
サービス率（μ：ミュー） → 単位時間あたりに処理できる要求の数
利用率（ρ：ロー） → サーバがどれだけ忙しいかの割合
利用率の公式：

ρ = λ / μ

例：毎秒3件到着（λ=3）、毎秒5件処理可能（μ=5）なら
→ ρ = 3/5 = 0.6（60%の利用率）

重要ポイント：

ρ < 1 でないとキューが無限に溜まる（破綻する）
ρが1に近づくほど待ち時間が急激に増える
4-8-2 待ち順序の計算と費用
待ち行列の順番やコストを計算する問題
処理時間が短いジョブを先に処理すると平均待ち時間が最小になる（SPT規則：最短処理時間優先）
4-8-3 利用率と平均待ち時間
M/M/1モデル（最も基本的なモデル）

M/M/1とは：

到着がランダム（ポアソン分布）
サービス時間がランダム（指数分布）
サーバが1台
平均待ち時間（Wq）の公式：

Wq = ρ / (μ ×（1 − ρ）)

平均応答時間（W）の公式：

W = 1 / (μ ×（1 − ρ）)

平均応答時間 ＝ 平均待ち時間 ＋ 平均サービス時間

4-8-4 M/M/Sモデルの平均待ち時間
サーバがS台ある場合のモデル
利用率 ρ = λ /（S × μ）
計算は複雑なので、試験では公式を与えられることが多い
4-8-5 CPU利用率と応答時間のグラフ
利用率が上がると応答時間は指数関数的に急増する
利用率70〜80%を超えると一気に遅くなる
だから余裕を持ったキャパシティプランニングが重要
計算が絡むので、問題で慣れていきましょう！

4-9 システムの信頼性特性と評価
4-9-1 システムの信頼性評価指標
MTBF（Mean Time Between Failures）

平均故障間隔。故障から次の故障までの平均時間
長いほど壊れにくい＝信頼性が高い
MTTR（Mean Time To Repair）

平均修理時間。故障してから復旧するまでの平均時間
短いほど早く直る＝保守性が高い
稼働率（可用性）

システムが正常に動いている割合
稼働率 = MTBF / (MTBF + MTTR)

例：MTBF=900時間、MTTR=100時間の場合 → 稼働率 = 900 / (900+100) = 900/1000 = 0.9（90%）

4-9-2 システムの信頼性計算
直列システム

すべてが正常でないと動かない構成
稼働率 = 各装置の稼働率の積
例：A(0.9) × B(0.9) = 0.81
並列システム

どれか1つでも正常なら動く構成
故障率（1−稼働率）の積を1から引く
例：1 −(1−0.9)×(1−0.9) = 1 − 0.01 = 0.99
4-9-3 複数システムの稼動率
直列と並列を組み合わせた構成の計算。手順は：

まず並列部分を1つの稼働率にまとめる
次に直列部分を掛け合わせる
4-9-4 通信網の構成と信頼性
ネットワークの経路が複数ある場合も、並列・直列の考え方で稼働率を計算します。

コラム：マイクロカーネルとモノリシックカーネル

OSの機能をすべてカーネル内に含む構造
高速だが、カーネルが大きく複雑
例：Linux
マイクロカーネル

カーネルには最小限の機能だけを持たせ、他はユーザモードで動かす
安定性・拡張性が高いが、オーバーヘッドが大きい

5-2 タスク（プロセス）管理
5-2-1 タスクとタスクの状態遷移
タスク（プロセス）とは、実行中のプログラムのことです。タスクには3つの状態があります：

実行状態（Running）

CPUを使って実際に処理を実行中の状態
実行可能状態（Ready）

CPUが空くのを待っている状態
いつでも実行できる準備はできている
待ち状態（Wait / Block）

入出力の完了など、イベント待ちの状態
CPUが空いても実行できない
状態遷移の流れ：

実行可能 →（ディスパッチ）→ 実行
実行 →（タイムアウトやプリエンプション）→ 実行可能
実行 →（入出力要求など）→ 待ち
待ち →（入出力完了など）→ 実行可能
重要： 待ち状態から直接実行状態にはならない！必ず実行可能状態を経由します。

5-2-4 同期制御
複数のタスクが協調して動作するための制御。あるタスクの処理結果を別のタスクが待つ、といった場面で必要です。

セマフォ

共有資源へのアクセスを制御する仕組み
P操作（wait）：資源を確保（カウンタを−1）
V操作（signal）：資源を解放（カウンタを+1）
5-2-5 排他制御
複数のタスクが同じ資源に同時にアクセスするのを防ぐ制御。

デッドロック

複数のタスクが互いに相手の持つ資源を待ち合って、どちらも先に進めなくなる状態
例：タスクAが資源1を持ち資源2を待つ、タスクBが資源2を持ち資源1を待つ → 永久に待ち続ける
5-2-2 タスクの切り替え
コンテキストスイッチ

実行中のタスクを切り替えること
現在のタスクの状態（レジスタの値など）を保存し、次のタスクの状態を復元する
切り替えにはオーバーヘッドがかかる
5-2-6 プロセスとスレッド
プロセス

独立したメモリ空間を持つ実行単位
プロセス間は互いに影響しにくい（安全）
スレッド

プロセス内の軽量な実行単位
同じプロセス内のスレッドはメモリを共有する
切り替えが高速だが、共有資源の競合に注意が必要
5-2-3 タスクのスケジューリング方式
CPUをどのタスクに割り当てるかを決める方式：

プリエンプティブ方式（横取り方式）

OSが強制的にタスクを切り替える
優先度の高いタスクが来たら、現在のタスクを中断
応答性が良い
ノンプリエンプティブ方式（非横取り方式）

タスクが自発的にCPUを明け渡すまで切り替わらない
実装が簡単だが応答性が悪い
主なスケジューリングアルゴリズム：

ラウンドロビン方式

各タスクに均等な**タイムクウォンタム（時間量子）**を割り当て、順番に実行
時間が来たら次のタスクに切り替え
公平だが、タイムクウォンタムが短すぎるとオーバーヘッド増大
優先度方式

優先度が高いタスクから先に実行
**飢餓（スタベーション）**の問題：優先度の低いタスクがいつまでも実行されない
多段フィードバックキュー方式

複数の優先度キューを用意
長く待ったタスクの優先度を上げて飢餓を防ぐ
内容が多いですが、どれも重要です。では問題いきます！

5-3 記憶管理
5-3-1 実記憶管理
物理的なメモリ（主記憶）をどう管理するかの話です。

固定区画方式

メモリをあらかじめ決まったサイズの区画に分割
プログラムを区画に割り当てる
実装が簡単だが、**内部フラグメンテーション（断片化）**が発生
内部フラグメンテーション＝区画よりプログラムが小さいと余った部分が無駄になる
可変区画方式

プログラムのサイズに合わせて動的に区画を確保
内部フラグメンテーションは起きない
ただし外部フラグメンテーションが発生
外部フラグメンテーション＝プログラムの割当・解放を繰り返すと隙間だらけになって大きなプログラムが入らない
コンパクション（メモリコンパクション）

外部フラグメンテーションの解消法
散らばったプログラムを詰め直して連続した空き領域を作る
処理コストが高い
5-3-2 仮想記憶管理
物理メモリより大きなプログラムを実行可能にする技術です。

補助記憶（ハードディスクやSSD）の一部を仮想的にメモリとして使う
プログラムは仮想アドレスを使い、OSが物理アドレスに変換する
必要な部分だけ主記憶に載せ、残りは補助記憶に置く
スワッピング

プロセス全体を主記憶と補助記憶の間で入れ替えること
スワップアウト → 主記憶→補助記憶に追い出す
スワップイン → 補助記憶→主記憶に読み込む
5-3-3 ページング方式
仮想記憶の代表的な実現方式です。

メモリを固定サイズのページに分割して管理
仮想メモリの単位＝ページ、物理メモリの単位＝ページフレーム
ページテーブルで仮想アドレスと物理アドレスの対応を管理
ページフォールト

必要なページが主記憶にない場合に発生する割込み
補助記憶からページを読み込む
ページ置き換えアルゴリズム（どのページを追い出すか）

FIFO（First In First Out）

最も古く読み込まれたページを追い出す
実装は簡単だが効率は良くない
LRU（Least Recently Used）

最も長い間使われていないページを追い出す
実用的で効率が良い
スラッシング

ページフォールトが頻発して、ページの入替ばかりで実際の処理がほとんど進まなくなる状態
主記憶が足りないときに発生する
コラム：メモリプール
あらかじめ一定量のメモリを確保しておき、必要に応じて割り当てる方式
メモリの確保・解放を高速に行える
コラム：リンク方式
プログラムの結合方式（静的リンク・動的リンク）

LRUの説明として正しいものはどれか？

A. 最も古く読み込まれたページを追い出す B. 最も長い間使われていないページを追い出す C. 最もサイズが大きいページを追い出す D. ランダムにページを追い出す

ブロックってなんだつけ

4月6日
ブロックは文脈によって意味が変わりますが：

ディスク（補助記憶）の場合

データを読み書きする最小単位
例：512バイトや4KBのブロック
キャッシュメモリの場合

キャッシュに読み込む単位を**ブロック（ライン）**と呼ぶ
ページとの関係

ページはメモリ管理の単位（仮想記憶の文脈）
ブロックはディスクやキャッシュの単位
ざっくり言うと：

メモリの話 → ページ
ディスクの話 → ブロック
キャッシュの話 → ブロック（ライン）
どれも「データを一定サイズに区切った塊」という点は同じです。呼び方が場面で変わるだけですね。

では問題6いきましょう！

LRUの説明として正しいものはどれか？

A. 最も古く読み込まれたページを追い出す B. 最も長い間使われていないページを追い出す C. 最もサイズが大きいページを追い出す D. ランダムにページを追い出す

c

4月6日

LRU＝Least Recently Used＝最も最近使われていないもの

FIFO → 最も古く読み込まれたページを追い出す（Aの説明）
LRU → 最も長い間使われていないページを追い出す
違いは：

FIFO → 「いつ入ってきたか」で判断
LRU → 「最後にいつ使ったか」で判断
LRUの方が実用的です。古くてもよく使うページは残すからですね。

5-4 言語処理ツール
5-4-1 言語プロセッサ
プログラム言語で書かれたコードを、コンピュータが実行できる形に変換するソフトウェアです。

アセンブラ

アセンブリ言語を機械語に変換する
1命令が1機械語にほぼ対応
コンパイラ

高水準言語（C、Javaなど）を機械語に一括変換する
変換後のプログラム＝オブジェクトコード（目的プログラム）
実行前にまとめて変換するので実行速度が速い
インタプリタ

ソースコードを1行ずつ解釈して即実行する
変換と実行を同時に行う
実行速度はコンパイラより遅いが、開発・デバッグが手軽
例：Python、JavaScript、Ruby
5-4-2 コンパイル技法
コンパイラがソースコードを変換する4つの段階：

① 字句解析

ソースコードを**トークン（最小単位の要素）**に分割
例：x = 10 + y → x, =, 10, +, y
② 構文解析

トークンの並びが文法的に正しいか**構文木（解析木）**を作って検証
③ 意味解析

型の整合性チェックなど、意味的に正しいかを検証
例：整数に文字列を足していないか
④ コード生成（最適化）

最終的な機械語コードを生成
不要な処理の削除や高速化の最適化も行う
5-4-3 リンク（連係編集）
コンパイルで生成されたオブジェクトコードを結合して実行可能ファイルにする作業です。

静的リンク

実行前にライブラリを組み込んで1つの実行ファイルにする
実行ファイルが大きくなるが、単体で動く
動的リンク

実行時に必要なライブラリをその都度読み込む
実行ファイルが小さい
ライブラリの更新が反映されやすい
**DLL（Dynamic Link Library）**が代表例
では問題いきます！

数字で表記：例 755 → 所有者rwx(7)、グループr-x(5)、その他r-x(5)
5-6-2 UNIX系OSの基本用語
シェル

ユーザとOSの仲介役
コマンドを受け取ってカーネルに伝える
例：bash、zsh、sh
カーネル

OSの中核。ハードウェアの管理を行う
パイプ（|）

あるコマンドの出力を次のコマンドの入力に渡す仕組み
例：ls | grep txt → ファイル一覧からtxtを含むものだけ表示
リダイレクト

コマンドの入出力先を変更する
> → 出力をファイルに書き込む（上書き）
>> → 出力をファイルに追記
< → ファイルから入力を受け取る
ワイルドカード

* → 任意の文字列（0文字以上）
? → 任意の1文字

4月9日
正解はBです。

6-1 データベース設計
6-1-1 関係データベースの設計
関係データベース（RDB）は**表（テーブル）**でデータを管理します。

基本用語

表（テーブル/リレーション） → データの集まり
行（レコード/タプル） → 1件分のデータ
列（フィールド/属性/カラム） → データの項目
主キー（プライマリキー） → 各行を一意に識別する列。重複・NULLは不可
外部キー（フォーリンキー） → 他のテーブルの主キーを参照する列。テーブル間の関連付けに使う
正規化

データの重複や矛盾を排除するためにテーブルを分割する作業です。

第1正規形

繰り返し項目を排除する
1つのセルに1つの値だけ（原子値）
第2正規形

第1正規形を満たしたうえで
部分関数従属を排除する
主キーの一部だけに従属する列を別テーブルに分離
第3正規形

第2正規形を満たしたうえで
推移的関数従属を排除する
主キー以外の列に従属する列を別テーブルに分離
6-1-2 データベース基礎における概念モデル
E-R図（Entity-Relationship図）

**エンティティ（実体）とリレーションシップ（関連）**でデータ構造を表現
エンティティ＝四角形、リレーションシップ＝ひし形や線
カーディナリティ（多重度）

1対1 → 社員と社員証
1対多 → 部署と社員（1つの部署に複数の社員）
多対多 → 学生と講義（学生は複数の講義を取り、講義には複数の学生）
6-1-3 インメモリデータベース
データを主記憶（メモリ）上に格納するデータベース
ディスクアクセスが不要なので非常に高速
ただし揮発性（電源を切るとデータが消える）なので、永続化の仕組みが必要
例：Redis、SAP HANA
